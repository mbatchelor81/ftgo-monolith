## 1. Executive Summary

The FTGO application is migrating from a single shared MySQL database (`ftgo`) to per-service
PostgreSQL databases as part of the microservices decomposition. Each service will own its
database schema, manage its own Flyway migrations, and use PostgreSQL IDENTITY columns for
ID generation instead of the shared `hibernate_sequence` table.

Cross-service foreign keys will be removed and replaced with soft references (stored IDs
without FK constraints), resolved at runtime via REST API calls or domain events.

### Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Database engine | PostgreSQL | Better JSON support, IDENTITY columns, modern ecosystem |
| Migration tool | Flyway (Spring Boot integrated) | Already in use; per-service migration sets |
| ID generation | `BIGINT GENERATED BY DEFAULT AS IDENTITY` | Per-service, no shared sequence table needed |
| Cross-service refs | Soft references (ID columns, no FK) | Service autonomy; resolved via API/events |
| Data sync | Event-driven + REST API fallback | Eventual consistency with compensating actions |

## 2. Current State: Shared Database

### Monolith Database: `ftgo` (MySQL)

All 7 tables reside in a single MySQL database with cross-table foreign keys:

```
ftgo (MySQL)
├── consumers                    (Consumer Service)
├── orders                       (Order Service)
│   ├── FK → courier.id          ← cross-service
│   └── FK → restaurants.id      ← cross-service
├── order_line_items             (Order Service)
│   └── FK → orders.id           ← intra-service
├── restaurants                  (Restaurant Service)
├── restaurant_menu_items        (Restaurant Service)
│   └── FK → restaurants.id      ← intra-service
├── courier                      (Courier Service)
├── courier_actions              (Courier Service)
│   ├── FK → courier.id          ← intra-service
│   └── FK → orders.id           ← cross-service
└── hibernate_sequence           (shared ID generation)
```

### Current Foreign Keys

| FK Constraint | Source Table | Target Table | Type |
|---------------|-------------|-------------|------|
| `orders_assigned_courier_id` | `orders.assigned_courier_id` | `courier.id` | **Cross-service** |
| `orders_restaurant_id` | `orders.restaurant_id` | `restaurants.id` | **Cross-service** |
| `courier_actions_order_id` | `courier_actions.order_id` | `orders.id` | **Cross-service** |
| `order_line_items_id` | `order_line_items.order_id` | `orders.id` | Intra-service |
| `courier_actions_courier_id` | `courier_actions.courier_id` | `courier.id` | Intra-service |
| `restaurant_menu_items_restaurant_id` | `restaurant_menu_items.restaurant_id` | `restaurants.id` | Intra-service |

### Problems with Shared Database

1. **Tight coupling**: Schema changes require coordinating all services
2. **Deployment risk**: A migration failure blocks all services
3. **Scaling bottleneck**: Single database limits horizontal scaling
4. **Technology lock-in**: All services must use the same DB engine and version
5. **Shared `hibernate_sequence`**: Contention under concurrent writes

## 4. Per-Service Schema Design

### 4.1 Consumer Service

**Database**: `ftgo_consumer_service`  
**Tables**: `consumers`

```sql
CREATE TABLE consumers (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    first_name  VARCHAR(255),
    last_name   VARCHAR(255),
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

**Notes**:
- Simplest schema — no cross-service references
- Consumer is a root aggregate
- `created_at` / `updated_at` added for auditability (not present in monolith)

### 4.2 Order Service

**Database**: `ftgo_order_service`  
**Tables**: `orders`, `order_line_items`

```sql
CREATE TABLE orders (
    id                          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    order_state                 VARCHAR(50) NOT NULL DEFAULT 'APPROVED',
    consumer_id                 BIGINT NOT NULL,        -- soft ref → consumer service
    restaurant_id               BIGINT NOT NULL,        -- soft ref → restaurant service
    assigned_courier_id         BIGINT,                 -- soft ref → courier service
    order_minimum               DECIMAL(19, 2),
    payment_token               VARCHAR(255),
    version                     BIGINT NOT NULL DEFAULT 0,
    delivery_address_street1    VARCHAR(255),
    delivery_address_street2    VARCHAR(255),
    delivery_address_city       VARCHAR(255),
    delivery_address_state      VARCHAR(255),
    delivery_address_zip        VARCHAR(255),
    accept_time                 TIMESTAMP,
    preparing_time              TIMESTAMP,
    ready_for_pickup_time       TIMESTAMP,
    picked_up_time              TIMESTAMP,
    delivered_time              TIMESTAMP,
    delivery_time               TIMESTAMP,
    ready_by                    TIMESTAMP,
    previous_ticket_state       INTEGER,
    created_at                  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at                  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE order_line_items (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    order_id        BIGINT NOT NULL REFERENCES orders (id) ON DELETE CASCADE,
    menu_item_id    VARCHAR(255),
    name            VARCHAR(255),
    price           DECIMAL(19, 2),
    quantity        INTEGER NOT NULL
);
```

**Key changes from monolith**:
- `consumer_id`, `restaurant_id`, `assigned_courier_id` are soft references (no FK)
- `order_line_items` gains its own `id` PK column (was FK-only in monolith)
- Intra-service FK between `order_line_items` → `orders` is retained with `ON DELETE CASCADE`
- Indexes on all cross-service reference columns for efficient joins/lookups

### 4.3 Restaurant Service

**Database**: `ftgo_restaurant_service`  
**Tables**: `restaurants`, `restaurant_menu_items`

```sql
CREATE TABLE restaurants (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name        VARCHAR(255) NOT NULL,
    street1     VARCHAR(255),
    street2     VARCHAR(255),
    city        VARCHAR(255),
    state       VARCHAR(255),
    zip         VARCHAR(255),
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE restaurant_menu_items (
    restaurant_id   BIGINT NOT NULL REFERENCES restaurants (id) ON DELETE CASCADE,
    id              VARCHAR(255) NOT NULL,
    name            VARCHAR(255),
    price           DECIMAL(19, 2),
    PRIMARY KEY (restaurant_id, id)
);
```

**Notes**:
- `restaurant_menu_items` uses a composite PK `(restaurant_id, id)` matching the `@ElementCollection` pattern
- Intra-service FK retained with `ON DELETE CASCADE`
- No cross-service references — Restaurant is a root aggregate

### 4.4 Courier Service

**Database**: `ftgo_courier_service`  
**Tables**: `courier`, `courier_actions`

```sql
CREATE TABLE courier (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    available   BOOLEAN NOT NULL DEFAULT FALSE,
    first_name  VARCHAR(255),
    last_name   VARCHAR(255),
    street1     VARCHAR(255),
    street2     VARCHAR(255),
    city        VARCHAR(255),
    state       VARCHAR(255),
    zip         VARCHAR(255),
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE courier_actions (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    courier_id  BIGINT NOT NULL REFERENCES courier (id) ON DELETE CASCADE,
    order_id    BIGINT,                 -- soft ref → order service
    time        TIMESTAMP,
    type        VARCHAR(255)
);
```

**Key changes from monolith**:
- `courier.available` changes from MySQL `BIT` to PostgreSQL `BOOLEAN`
- `courier_actions.order_id` is a soft reference (no FK to orders)
- `courier_actions` gains its own `id` PK column
- Intra-service FK between `courier_actions` → `courier` is retained

## 5. Cross-Service Foreign Key Removal Plan

### Phase 1: Add Soft Reference Columns (Parallel Operation Period)

During the transition, both monolith and microservices will run simultaneously.
Cross-service FKs in the monolith will remain until the monolith is decommissioned.

The per-service databases are created **without** cross-service FKs from the start.

### Phase 2: Resolve References at Runtime

| Cross-Service Reference | Resolution Strategy |
|------------------------|---------------------|
| `orders.consumer_id` → Consumer Service | REST API: `GET /consumers/{id}` — called during `createOrder` for validation |
| `orders.restaurant_id` → Restaurant Service | REST API: `GET /restaurants/{id}` — called during order creation for menu validation |
| `orders.assigned_courier_id` → Courier Service | Event-driven: Courier assignment published as domain event |
| `courier_actions.order_id` → Order Service | Event-driven: Order lifecycle events consumed by Courier Service |

### Phase 3: Data Validation

Without cross-service FKs, referential integrity is enforced at the application layer:

1. **On write**: Validate referenced entity exists via synchronous API call before persisting
2. **On read**: Handle missing references gracefully (entity may have been deleted in source service)
3. **Periodic audit**: Background job compares cross-service IDs to detect orphaned references

### Orphaned Reference Handling

```
┌─ Order Service ─────────────────────────────────┐
│                                                   │
│  createOrder(consumerId, restaurantId):            │
│    1. Call Consumer Service → validate consumer    │
│    2. Call Restaurant Service → validate restaurant│
│    3. If either fails → reject order               │
│    4. Persist order with soft references           │
│                                                   │
│  getOrder(orderId):                                │
│    1. Load order from local DB                     │
│    2. Optionally enrich with consumer/restaurant   │
│       data via API calls (or use cached data)      │
│                                                   │
└───────────────────────────────────────────────────┘
```

## 6. ID Generation Strategy

### Problem: Shared `hibernate_sequence`

The monolith uses a single `hibernate_sequence` table for ID generation across all entities.
This creates:
- Contention under concurrent writes
- Implicit coupling between services
- Non-contiguous IDs per entity type

### Solution: PostgreSQL IDENTITY Columns

Each service uses PostgreSQL `GENERATED BY DEFAULT AS IDENTITY` for primary keys:

```sql
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

**Why `BY DEFAULT` instead of `ALWAYS`**:
- `BY DEFAULT` allows explicit ID insertion during data migration from the monolith
- After migration completes, existing IDs from the monolith are preserved
- New records use auto-generated IDs from the service-local sequence

### ID Mapping During Migration

During the parallel operation period, IDs must be consistent between monolith and
microservice databases:

1. **Export** data from monolith MySQL with existing IDs
2. **Import** into per-service PostgreSQL databases with explicit IDs
3. **Reset** the IDENTITY sequence to `MAX(id) + 1` after import:

```sql
-- Example for orders table after data import
SELECT setval(pg_get_serial_sequence('orders', 'id'), (SELECT MAX(id) FROM orders));
```

### Future Considerations: Globally Unique IDs

If services need globally unique IDs in the future (e.g., for event sourcing or distributed
tracing), consider:
- **UUIDs**: Simple but larger storage footprint
- **Snowflake IDs**: Twitter-style 64-bit IDs with embedded timestamp + service ID
- **ULID**: Sortable, 128-bit, lexicographically ordered

For now, `BIGINT IDENTITY` is sufficient since IDs are only unique within each service.

## 7. Flyway Migration Structure and Naming Conventions

### Directory Structure

```
services/
├── ftgo-consumer-service/
│   └── src/main/resources/db/migration/
│       └── V1__create_consumer_service_schema.sql
├── ftgo-order-service/
│   └── src/main/resources/db/migration/
│       └── V1__create_order_service_schema.sql
├── ftgo-restaurant-service/
│   └── src/main/resources/db/migration/
│       └── V1__create_restaurant_service_schema.sql
└── ftgo-courier-service/
    └── src/main/resources/db/migration/
        └── V1__create_courier_service_schema.sql
```

### Naming Convention

```
V{version}__{description}.sql     — Versioned migrations (schema changes)
R__{description}.sql              — Repeatable migrations (views, functions, seed data)
```

**Version numbering**: Sequential integers starting at `V1` per service.  
**Description**: Snake_case, descriptive of the change.

Examples:
```
V1__create_consumer_service_schema.sql
V2__add_consumer_email_column.sql
V3__create_consumer_preferences_table.sql
R__refresh_consumer_statistics_view.sql
```

### Spring Boot Flyway Configuration

Each service's `application.yml` configures Flyway:

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
    baseline-on-migrate: true
    baseline-version: 0
    schemas: public
    validate-migration-naming: true
```

**Key settings**:
- `baseline-on-migrate: true` — allows Flyway to manage existing databases
- `baseline-version: 0` — baseline at version 0 so V1 is the first real migration
- `validate-migration-naming: true` — enforce naming convention compliance
- `locations: classpath:db/migration` — per-service migration directory

### Migration Authoring Rules

1. **Each migration must be idempotent** where possible (use `IF NOT EXISTS`)
2. **Never modify a migration** after it has been applied to any environment
3. **One logical change per migration** — don't combine unrelated schema changes
4. **Always include rollback comments** — document how to undo the migration
5. **Test migrations** against a clean database and against the previous version
6. **No DML in versioned migrations** — use repeatable migrations for seed data

## 8. Data Synchronization Approach

### Strategy: Event-Driven with REST API Fallback

Data synchronization between services uses a hybrid approach:

```
┌──────────────┐    Domain Events     ┌──────────────┐
│  Order Svc   │ ──────────────────►  │  Courier Svc │
│              │                      │              │
│  (publishes  │    REST API Call      │  (consumes   │
│   OrderCreated,│ ◄──────────────── │   events to  │
│   OrderAccepted│                    │   update plan)│
│   etc.)      │                      │              │
└──────────────┘                      └──────────────┘
        │                                     ▲
        │ REST API Call                       │
        ▼                                     │
┌──────────────┐                      ┌──────────────┐
│ Consumer Svc │                      │Restaurant Svc│
│              │                      │              │
│ (validates   │                      │ (provides    │
│  consumer on │                      │  menu data   │
│  order create)│                     │  via REST)   │
└──────────────┘                      └──────────────┘
```

### Synchronization Patterns

#### Pattern 1: Synchronous REST API Call (Query)

**Use when**: Data is needed immediately and must be fresh.

| Caller | Target | Endpoint | When |
|--------|--------|----------|------|
| Order Service | Consumer Service | `GET /consumers/{id}` | Order creation — validate consumer |
| Order Service | Restaurant Service | `GET /restaurants/{id}` | Order creation — validate restaurant, get menu |

**Implementation**:
```java
// In Order Service
@Service
public class OrderService {
    private final ConsumerServiceClient consumerClient;
    private final RestaurantServiceClient restaurantClient;

    public Order createOrder(CreateOrderRequest request) {
        // Validate consumer exists (synchronous call)
        consumerClient.validateConsumer(request.getConsumerId());

        // Validate restaurant and get menu (synchronous call)
        Restaurant restaurant = restaurantClient.getRestaurant(request.getRestaurantId());

        // Create order with soft references
        Order order = new Order(request.getConsumerId(), request.getRestaurantId(), ...);
        return orderRepository.save(order);
    }
}
```

**Resilience**: Use circuit breakers (Resilience4j) and fallback caches to handle target service unavailability.

#### Pattern 2: Domain Events (Asynchronous)

**Use when**: Other services need to react to state changes but don't need immediate consistency.

| Publisher | Event | Consumers | Action |
|-----------|-------|-----------|--------|
| Order Service | `OrderCreated` | Courier Service | Schedule potential delivery |
| Order Service | `OrderAccepted` | Courier Service | Assign courier, create actions |
| Order Service | `OrderCancelled` | Courier Service | Cancel delivery actions |
| Restaurant Service | `MenuUpdated` | Order Service | Invalidate cached menu data |
| Courier Service | `CourierAssigned` | Order Service | Update `assigned_courier_id` |

**Implementation approach**: Spring Application Events initially (in-process), migrating to a message broker (e.g., Apache Kafka, RabbitMQ) when services are fully separated.

#### Pattern 3: Local Cache with Event Invalidation

**Use when**: Frequently accessed reference data that changes infrequently.

Example: Order Service caches restaurant menu data locally.

```
Restaurant Service ──MenuUpdated event──► Order Service (invalidate cache)
                                           │
                    ◄──GET /restaurants/{id}─┘ (cache miss → fetch fresh)
```

### Data Consistency Model

**Eventual consistency** is the default model:
- Cross-service data may be temporarily stale
- Compensating actions handle failures (see Section 11)
- Idempotent event handlers ensure safe reprocessing

## 9. Migration Runbook

### Prerequisites

- [ ] Per-service PostgreSQL databases created
- [ ] Service application.yml configured with correct JDBC URLs
- [ ] Flyway migrations tested in staging environment
- [ ] Data export scripts prepared and tested
- [ ] Rollback scripts prepared and tested
- [ ] Monitoring and alerting configured for all service databases

### Phase 1: Provision Per-Service Databases

```bash
# Create databases (run as PostgreSQL superuser)
CREATE DATABASE ftgo_consumer_service OWNER ftgo;
CREATE DATABASE ftgo_order_service OWNER ftgo;
CREATE DATABASE ftgo_restaurant_service OWNER ftgo;
CREATE DATABASE ftgo_courier_service OWNER ftgo;
```

### Phase 2: Run Flyway Migrations

```bash
# Each service runs its own migrations on startup, or manually:
cd services/ftgo-consumer-service
../gradlew flywayMigrate

cd services/ftgo-order-service
../gradlew flywayMigrate

cd services/ftgo-restaurant-service
../gradlew flywayMigrate

cd services/ftgo-courier-service
../gradlew flywayMigrate
```

### Phase 3: Export Data from Monolith

```bash
# Export each table from the monolith MySQL database
mysqldump -u root ftgo consumers --no-create-info --complete-insert > consumers_data.sql
mysqldump -u root ftgo orders --no-create-info --complete-insert > orders_data.sql
mysqldump -u root ftgo order_line_items --no-create-info --complete-insert > order_line_items_data.sql
mysqldump -u root ftgo restaurants --no-create-info --complete-insert > restaurants_data.sql
mysqldump -u root ftgo restaurant_menu_items --no-create-info --complete-insert > restaurant_menu_items_data.sql
mysqldump -u root ftgo courier --no-create-info --complete-insert > courier_data.sql
mysqldump -u root ftgo courier_actions --no-create-info --complete-insert > courier_actions_data.sql
```

### Phase 4: Transform and Import Data

Data transformation is needed to convert MySQL syntax to PostgreSQL:

1. **Boolean conversion**: MySQL `BIT(1)` → PostgreSQL `BOOLEAN`
2. **Datetime conversion**: MySQL `DATETIME` → PostgreSQL `TIMESTAMP`
3. **Remove MySQL-specific syntax**: `ENGINE=InnoDB`, backtick quoting

```bash
# Example: Import consumers into Consumer Service database
psql -U ftgo -d ftgo_consumer_service -f consumers_data_pg.sql

# Reset IDENTITY sequences after import
psql -U ftgo -d ftgo_consumer_service -c \
  "SELECT setval(pg_get_serial_sequence('consumers', 'id'), (SELECT COALESCE(MAX(id), 0) FROM consumers));"

psql -U ftgo -d ftgo_order_service -c \
  "SELECT setval(pg_get_serial_sequence('orders', 'id'), (SELECT COALESCE(MAX(id), 0) FROM orders));"

psql -U ftgo -d ftgo_restaurant_service -c \
  "SELECT setval(pg_get_serial_sequence('restaurants', 'id'), (SELECT COALESCE(MAX(id), 0) FROM restaurants));"

psql -U ftgo -d ftgo_courier_service -c \
  "SELECT setval(pg_get_serial_sequence('courier', 'id'), (SELECT COALESCE(MAX(id), 0) FROM courier));"
```

### Phase 5: Validate Data

```sql
-- Compare row counts between monolith and per-service databases
-- Monolith (MySQL)
SELECT 'consumers' as tbl, COUNT(*) FROM consumers
UNION ALL SELECT 'orders', COUNT(*) FROM orders
UNION ALL SELECT 'order_line_items', COUNT(*) FROM order_line_items
UNION ALL SELECT 'restaurants', COUNT(*) FROM restaurants
UNION ALL SELECT 'restaurant_menu_items', COUNT(*) FROM restaurant_menu_items
UNION ALL SELECT 'courier', COUNT(*) FROM courier
UNION ALL SELECT 'courier_actions', COUNT(*) FROM courier_actions;

-- Per-service databases (PostgreSQL) — run against each DB
-- psql -d ftgo_consumer_service -c "SELECT COUNT(*) FROM consumers;"
-- psql -d ftgo_order_service -c "SELECT COUNT(*) FROM orders;"
-- ... etc.
```

### Phase 6: Parallel Operation

1. Deploy microservices alongside the monolith
2. Route new traffic to microservices
3. Monitor for data consistency issues
4. Run cross-service reference audit jobs

### Phase 7: Decommission Monolith Database

1. Stop all monolith write traffic
2. Final data sync from monolith → per-service databases
3. Verify data completeness
4. Decommission monolith MySQL database
5. Remove `ftgo-flyway` module from build

## 10. Rollback Strategy

### Per-Migration Rollback

Each Flyway versioned migration should have a corresponding undo script documented
in comments at the top of the migration file. Flyway Community Edition does not support
automatic undo, so rollbacks are manual.

**Convention**: For each `V{n}__{description}.sql`, maintain a documented rollback
procedure in the migration file header comments.

Example:
```sql
-- Rollback: DROP TABLE IF EXISTS consumers;
```

### Full Service Rollback

If a service database migration fails catastrophically:

1. **Stop the service** to prevent further writes
2. **Restore from backup** (PostgreSQL `pg_restore` from latest `pg_dump`)
3. **Reset Flyway history**: `DELETE FROM flyway_schema_history WHERE version = '{failed_version}';`
4. **Fix the migration** and redeploy
5. **Restart the service**

### Database Backup Schedule

| Environment | Frequency | Retention | Method |
|-------------|-----------|-----------|--------|
| Production | Every 6 hours | 30 days | `pg_dump` + cloud storage |
| Staging | Daily | 7 days | `pg_dump` + cloud storage |
| Development | On-demand | 3 days | Local `pg_dump` |

### Point-in-Time Recovery

Enable PostgreSQL WAL archiving for point-in-time recovery in production:

```yaml
# postgresql.conf
wal_level = replica
archive_mode = on
archive_command = 'cp %p /archive/%f'
```

## 11. Data Consistency Strategy

### Saga Pattern for Cross-Service Operations

The most critical cross-service operation is **order creation**, which involves:
1. Consumer Service: Validate consumer
2. Restaurant Service: Validate restaurant and menu
3. Order Service: Create order
4. Courier Service: Schedule delivery

This is implemented as a **choreography-based saga**:

```
Consumer Svc          Order Svc           Restaurant Svc      Courier Svc
     │                    │                     │                  │
     │◄── validate ───────│                     │                  │
     │─── OK ────────────►│                     │                  │
     │                    │──── validate ───────►│                  │
```

### Compensating Actions

If any step in the saga fails, compensating actions undo previous steps:

| Failure Point | Compensating Action |
|--------------|---------------------|
| Consumer validation fails | Reject order — no compensation needed |
| Restaurant validation fails | Reject order — no compensation needed |
| Order creation fails | No side effects to undo |
| Courier assignment fails | Mark order as `PENDING_COURIER` — retry with backoff |

### Idempotency Requirements

All event handlers and API endpoints must be idempotent:

- **Event handlers**: Use event ID as idempotency key; track processed events
- **REST APIs**: Use request ID header for write operations
- **Database operations**: Use `INSERT ... ON CONFLICT DO NOTHING` where appropriate

### Monitoring and Alerting

| Metric | Alert Threshold | Action |
|--------|----------------|--------|
| Cross-service API latency | p99 > 500ms | Investigate target service |
| Event processing lag | > 1 minute | Scale consumers / check dead letter queue |
| Orphaned reference count | > 0 (daily audit) | Investigate and reconcile |
| Saga failure rate | > 1% | Review compensating actions |

## 12. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Data loss during migration | Low | Critical | Pre-migration backup + row count validation |
| ID collision after migration | Low | High | Reset IDENTITY sequences to `MAX(id) + 1` |
| Cross-service call failures | Medium | Medium | Circuit breakers + fallback caches |
| Eventual consistency confusion | Medium | Medium | Clear documentation + monitoring |
| Flyway migration ordering issues | Low | Medium | Per-service versioning + CI validation |
| Performance degradation from API calls | Medium | Medium | Caching + async event-driven patterns |
| Rollback complexity | Low | High | Automated backup + tested restore procedures |
