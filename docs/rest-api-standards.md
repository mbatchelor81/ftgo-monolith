# FTGO REST API Standards

> **Status**: Accepted — EM-45
> **Scope**: Every FTGO microservice under `services/*` and every internal API
> exposed by the legacy `ftgo-*-service` modules until they migrate.
>
> This document defines the conventions every FTGO REST API must follow so
> that consumers (mobile apps, the courier web UI, other internal services,
> and external partners) see a consistent surface.

---

## 1. API Documentation (OpenAPI 3.0)

- Every service **MUST** publish a machine-readable OpenAPI 3.0 document.
- The spec is generated by [SpringDoc][springdoc] (the replacement for the
  deprecated Springfox Swagger 2.x toolchain, see [ADR 0002](adr/0002-openapi-springdoc.md)).
- Services get SpringDoc "for free" by depending on the shared
  [`libs:ftgo-openapi`](../libs/ftgo-openapi/) library.
- The following endpoints **MUST** be exposed on every service:

  | Endpoint            | Purpose                              |
  |---------------------|--------------------------------------|
  | `/v3/api-docs`      | Raw OpenAPI 3.0 JSON                 |
  | `/v3/api-docs.yaml` | Raw OpenAPI 3.0 YAML                 |
  | `/swagger-ui.html`  | Interactive Swagger UI browser       |

- Per-service metadata (title, description, contact, licence) is configured
  via the `ftgo.openapi.*` keys in `application.yml` — see
  [`OpenApiProperties`](../libs/ftgo-openapi/src/main/java/net/chrisrichardson/ftgo/openapi/OpenApiProperties.java).

[springdoc]: https://springdoc.org/

## 2. URL Naming

- **Plural, lowercase, kebab-case nouns** for resources.
  - Good: `/orders`, `/consumer-profiles`
  - Bad: `/Order`, `/consumerProfiles`, `/getOrder`
- **Nested resources** reflect ownership: `/restaurants/{id}/menu-items`.
- **Hierarchy depth SHOULD NOT exceed two levels** (`/a/{id}/b/{id}`); beyond
  that, expose the inner resource as a top-level endpoint with a query
  parameter (`/menu-items?restaurantId={id}`).
- **No verbs in path segments.** Use HTTP methods to express the action.
  - Good: `POST /orders`
  - Bad: `POST /createOrder`, `GET /getOrders`
- **Exception — state transitions**: "action sub-resources" are allowed when
  the transition is not idempotent and has no natural `PATCH` equivalent:
  `POST /orders/{id}/cancel`, `POST /orders/{id}/accept`.

## 3. HTTP Methods

| Method  | Purpose                        | Idempotent | Request body | Response body |
|---------|--------------------------------|:----------:|:------------:|:-------------:|
| `GET`   | Read (single or collection)    |     Yes    |      No      |      Yes      |
| `POST`  | Create, or non-idempotent action |    No    |     Yes      |   Usually     |
| `PUT`   | Full replacement of a resource |     Yes    |     Yes      |   Optional    |
| `PATCH` | Partial update                 |     Yes    |     Yes      |   Optional    |
| `DELETE`| Remove a resource              |     Yes    |   Optional   |   Optional    |

## 4. HTTP Status Codes

| Code | When to use                                                              |
|------|--------------------------------------------------------------------------|
| 200 OK                    | Successful `GET`, `PUT`, `PATCH`, or `DELETE` that returns a body. |
| 201 Created               | Successful `POST` that creates a resource. MUST include a `Location` header. |
| 202 Accepted              | The request was accepted for async processing.                    |
| 204 No Content            | Successful request with no response body (typical for `DELETE`).  |
| 400 Bad Request           | Validation error in the request payload or query parameters.     |
| 401 Unauthorized          | Missing or invalid authentication credentials.                   |
| 403 Forbidden             | Authenticated user is not permitted to perform the action.       |
| 404 Not Found             | The identified resource does not exist.                          |
| 409 Conflict              | Invalid state transition (e.g. cancelling a `DELIVERED` order).  |
| 422 Unprocessable Entity  | Request is syntactically valid but semantically rejected.        |
| 429 Too Many Requests     | Rate limit exceeded.                                             |
| 500 Internal Server Error | Unhandled exception. Never leak stack traces.                    |
| 503 Service Unavailable   | Downstream dependency unavailable; retry-safe.                   |

### Error payload ([RFC 7807 Problem Details][rfc7807])

```json
{
  "type":     "https://api.ftgo.example.com/errors/order-not-found",
  "title":    "Order not found",
  "status":   404,
  "detail":   "No order exists with id 42.",
  "instance": "/api/v1/orders/42",
  "timestamp": "2026-04-21T17:30:00Z"
}
```

[rfc7807]: https://datatracker.ietf.org/doc/html/rfc7807

## 5. Pagination

All collection-returning endpoints **MUST** paginate.

### Query parameters

| Name   | Type | Default | Max   | Description                                                                          |
|--------|------|:-------:|:-----:|--------------------------------------------------------------------------------------|
| `page` | int  |   `0`   |   —   | Zero-indexed page number.                                                            |
| `size` | int  |  `20`   | `100` | Page size. Requests exceeding the max are clamped and a warning is logged.           |
| `sort` | str  |   —     |   —   | `field,asc` or `field,desc`. Multiple `sort=` parameters are applied in order.        |

### Response envelope

```json
{
  "content": [ { "...": "..." } ],
  "page": {
    "number": 0,
    "size": 20,
    "totalElements": 137,
    "totalPages": 7
  },
  "_links": {
    "self":  { "href": "/api/v1/orders?page=0&size=20" },
    "next":  { "href": "/api/v1/orders?page=1&size=20" },
    "last":  { "href": "/api/v1/orders?page=6&size=20" }
  }
}
```

Clients **MUST** follow `_links.next` / `_links.prev` rather than constructing
page URLs themselves, so the server retains the freedom to migrate to cursor
pagination without breaking callers.

## 6. Date & Time Format

- All date-time values **MUST** be serialised as **ISO 8601** strings in
  **UTC** with a trailing `Z`: `2026-04-21T17:30:00Z`.
- Date-only values use `YYYY-MM-DD` with no time component.
- Services **MUST NOT** emit numeric epoch timestamps on the wire — they
  obscure the timezone and are not self-describing.
- Jackson is configured in the `ftgo-common` library to (de)serialise
  `java.time.Instant`, `OffsetDateTime`, and `LocalDate` in these formats
  automatically; services should **prefer `Instant` over `Date`** on their
  DTOs.

## 7. Content Negotiation

- Default media type is `application/json; charset=utf-8`.
- Requests with `Accept: application/problem+json` receive errors formatted
  per RFC 7807.
- Services **MAY** support `application/yaml` for configuration-heavy
  resources but **MUST** also accept `application/json`.

## 8. Naming of JSON Fields

- `camelCase` for all property names (`consumerId`, not `consumer_id`).
- Boolean properties read naturally: `isAvailable`, `hasBeenPaid`.
- Money values are objects, not numbers: `{ "amount": "12.50", "currency": "USD" }`.
  The string representation avoids floating-point rounding.
- Enumerations are `UPPER_SNAKE_CASE` strings, matching the Java `enum`
  constant name (e.g. `READY_FOR_PICKUP`).

## 9. Validation

- Every incoming DTO **MUST** be annotated with Jakarta Bean Validation
  constraints (`@NotNull`, `@Size`, `@Pattern`, `@Positive`, …).
- Controllers **MUST** add `@Valid` to the request body parameter so that
  Spring's `MethodArgumentNotValidException` fires on bad input.
- A `@RestControllerAdvice` in each service converts validation failures into
  RFC 7807 `400 Bad Request` responses.

## 10. Security

- Production endpoints **MUST** be protected by a bearer JWT (see EM-40).
  - The shared OpenAPI configuration pre-registers a `bearer-jwt` security
    scheme — controllers opt in per-endpoint with
    `@SecurityRequirement(name = "bearer-jwt")`.
- `/actuator/**`, `/v3/api-docs/**`, and `/swagger-ui.html` are public in
  non-production profiles and behind authentication (or network ACL) in
  production.
- CORS is configured per-service; wildcard (`*`) origins are forbidden in
  production.

## 11. Annotations Cheat Sheet (SpringDoc)

```java
@Tag(name = "Orders", description = "Place, revise, and track customer orders.")
@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {

  @Operation(
      summary     = "Create a new order",
      description = "Creates an APPROVED order for the given consumer and restaurant.")
  @ApiResponses({
      @ApiResponse(responseCode = "201", description = "Order created"),
      @ApiResponse(responseCode = "400", description = "Validation error",
                   content = @Content(schema = @Schema(implementation = ProblemDetail.class))),
      @ApiResponse(responseCode = "404", description = "Consumer or restaurant not found")
  })
  @PostMapping
  public ResponseEntity<CreateOrderResponse> create(@Valid @RequestBody CreateOrderRequest req) { … }
}
```

See the [SpringDoc reference documentation][springdoc-docs] for the full list
of annotations.

[springdoc-docs]: https://springdoc.org/#migrating-from-springfox

## 12. Checklist for Every New Endpoint

- [ ] URL follows §2 (plural, kebab-case, versioned).
- [ ] HTTP method matches §3.
- [ ] Returns the right status code per §4.
- [ ] Paginates collections per §5.
- [ ] Uses ISO 8601 UTC for every date-time.
- [ ] Request body is validated with Jakarta Bean Validation.
- [ ] Annotated with `@Operation`, `@ApiResponses`, and (where applicable)
      `@SecurityRequirement`.
- [ ] Appears in Swagger UI at `/swagger-ui.html`.
- [ ] Unit + integration tests cover the happy path and at least one error
      case.
