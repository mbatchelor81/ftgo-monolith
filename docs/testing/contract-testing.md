# Contract Testing for FTGO Microservices

Contract tests are the FTGO answer to "how do we change an API without
breaking the five services that call it?". They sit between
**integration** and **API** tests on the pyramid
(see [`docs/testing-strategy.md`](../testing-strategy.md) §5) and
catch cross-service regressions at PR-time — before a bad merge can
reach a downstream consumer.

## 1. Tooling

FTGO uses **Spring Cloud Contract** for every intra-FTGO call.
Rationale:

- Ships a Gradle plugin with verifier codegen; zero runtime glue.
- Producer-owned contracts work well for our monorepo (each service
  owns one directory of contract definitions).
- WireMock stubs generated by the plugin can be consumed by any HTTP
  client — not just Spring's `RestTemplate`.
- Extends smoothly to messaging contracts (Kafka, RabbitMQ) once the
  event-driven rollout starts.

Pact is still acceptable for **external** consumers (partner APIs,
mobile clients) where the contract must be authored by the consumer.

## 2. Who owns what

- **Producer (the service exposing the API)** owns:
  - Contract definitions (Groovy DSL) under
    `src/test/resources/contracts/<consumer>/…`.
  - A `BaseContract` test class that bootstraps the Spring context the
    verifier runs against.
  - Publishing stubs via the Spring Cloud Contract plugin.
- **Consumer (the service calling the API)** owns:
  - `@AutoConfigureStubRunner` wiring that downloads the producer's
    stubs and starts WireMock on the consumer classpath.
  - Its own consumer-driven expectations (e.g. "I need field `x`").

## 3. Producer side — setup

### `build.gradle` snippet

Add to the producing service's `build.gradle` (e.g.
`services/order-service/build.gradle`):

```groovy
plugins {
    id 'org.springframework.cloud.contract' version '4.1.1'
}

dependencies {
    testImplementation 'org.springframework.cloud:spring-cloud-starter-contract-verifier'
}

contracts {
    baseClassForTests = 'com.ftgo.order.contract.BaseContract'
    packageWithBaseClasses = 'com.ftgo.order.contract'
    contractsDslDir = file('src/test/resources/contracts')
}
```

### Contract DSL

```groovy
// src/test/resources/contracts/consumer-service/shouldReturnOrderById.groovy
package contracts.consumer

import org.springframework.cloud.contract.spec.Contract

Contract.make {
    description "should return an order by id"
    request {
        method GET()
        url "/orders/99"
        headers { contentType applicationJson() }
    }
    response {
        status OK()
        headers { contentType applicationJson() }
        body([
            orderId   : 99,
            state     : "APPROVED",
            orderTotal: "62.50",
            consumerId: 1511300065921
        ])
    }
}
```

### Base test class

```java
package com.ftgo.order.contract;

import com.ftgo.order.web.OrderController;
import io.restassured.module.mockmvc.RestAssuredMockMvc;
import org.junit.jupiter.api.BeforeEach;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;

@SpringBootTest
public abstract class BaseContract {

    @Autowired OrderController orderController;

    @BeforeEach
    void setUp() {
        RestAssuredMockMvc.standaloneSetup(orderController);
    }
}
```

Running `./gradlew :services:order-service:generateContractTests`
generates `ContractVerifierTest` classes that fail if the live
controller doesn't honour every contract.

Stubs are published to the module's `build/stubs/` and uploaded as
CI artifacts by the service's workflow.

## 4. Consumer side — setup

### `build.gradle`

```groovy
dependencies {
    testImplementation 'org.springframework.cloud:spring-cloud-starter-contract-stub-runner'
}
```

### Test class

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.NONE)
@AutoConfigureStubRunner(
    ids = "com.ftgo.order:order-service:+:stubs:8090",
    stubsMode = StubRunnerProperties.StubsMode.LOCAL
)
class OrderServiceClientContractTest {

    @Autowired OrderServiceClient client;

    @Test
    void fetchOrder_returnsStubbedFixture() {
        OrderResponse order = client.fetchOrder(99L);

        assertThat(order.state()).isEqualTo("APPROVED");
        assertThat(order.orderTotal()).isEqualTo("62.50");
    }
}
```

`stubsMode = LOCAL` uses the Maven local repository (fast, works
offline). CI switches this to `REMOTE` once a stubs registry is wired
up.

## 5. Naming and layout

```
services/<context>-service/
└── src/test/
    ├── java/com/ftgo/<context>/
    │   └── contract/
    │       └── BaseContract.java             ← one per service
    └── resources/contracts/
        ├── consumer-service/
        │   └── shouldXxx.groovy
        └── courier-service/
            └── shouldXxx.groovy
```

One subdirectory per **consumer**. The directory name appears in
generated method names, so keep it matching the service directory
(e.g. `consumer-service`, not `Consumer Service`).

## 6. Versioning

- Contracts are versioned alongside the producer artifact — the stub
  JAR version matches the producing service's published version.
- Breaking changes (removed fields, changed types) require:
  1. A new contract capturing the new shape.
  2. Explicit acknowledgement in the PR description (`BREAKING:` tag).
  3. Consumer-side PRs bumping to the new stubs version.
- Additive changes (new optional fields, new endpoints) do not break
  downstream consumers and can be merged without coordination.

## 7. What contract tests do **not** cover

- Authorization rules — tested as integration tests with `@WithMockUser`.
- Rate limiting, CORS, observability — tested as API tests.
- Cross-service **workflows** — covered by E2E tests in
  `ftgo-end-to-end-tests/`.
